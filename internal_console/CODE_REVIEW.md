# CODE_REVIEW.md

## 1. 文件定位
本文件用于约束 `internal_console/` 的前端代码审查标准、页面实现质量要求、契约约束和当前阶段应拒绝的改动类型。

---

## 2. 审查目标
前端代码审查应优先保证：

1. 页面职责清楚
2. API 调用边界清楚
3. SSE 行为处理正确
4. 错误状态可见
5. 代码结构可持续演进
6. 不把后端业务逻辑错误搬到前端硬编码

---

## 3. 全局审查原则

### 3.1 页面不要承担后端业务逻辑
前端页面只能：
- 输入参数
- 调后端 API
- 展示结果
- 管理交互状态

不能把以下内容硬编码进前端：
- retrieval 业务规则
- citation 生成规则
- build 判定逻辑
- 评估打分逻辑
- memory 策略

### 3.2 API-first
所有页面必须通过稳定 API 获取数据。  
如果页面需要的数据当前后端没有提供，应先补后端 API，而不是前端绕过边界。
页面与 API client 应按 chat / knowledge / evaluation / runtime 等领域拆分，不延续 `*_console` 风格命名。

### 3.3 前后端参数一致性必须审查
代码评审时必须检查前端 request 参数链路是否与后端 contract 完整一致。

至少要核对：
- `types/` 中的 request 类型是否覆盖后端字段
- `api/` 中的 client 是否透传全部必要参数
- `features/` 中的 hook 是否维护了对应状态
- `pages/` 中的表单是否提供了对应输入项或明确的默认策略

以下情况应视为缺陷：
- 后端已公开字段，页面没有输入项
- API client 支持字段，但 hook 未传递
- 页面有输入项，但 API client 未透传
- 通过前端硬编码默认值掩盖契约遗漏

### 3.4 SSE 处理必须按事件语义实现
对 `/chat_stream`：
- 不允许把所有事件都当普通字符串流处理
- 必须正确处理 started / delta / heartbeat / completed / cancelled / error
- cancel 场景必须能正确收口 UI 状态

### 3.5 错误必须可见
禁止：
- 请求失败只在控制台打印，不在页面可见
- stream 失败无状态反馈
- evaluation/build 失败后页面无明显提示

### 3.5 类型必须明确
前端 contract 类型必须明确维护。  
不允许大量使用 `any` 规避类型约束。

## 4. 页面级审查点

## 4.1 Chat Playground
审查重点：
- `/chat` 与 `/chat_stream` 路径区分清楚
- SSE 生命周期处理正确
- cancel 行为正确
- citations 展示与 completed 状态联动正确
- metadata / trace 摘要展示不混乱

## 4.2 Knowledge Ingest
审查重点：
- 上传与构建操作反馈明确
- 构建失败状态明确
- 结果摘要清晰
- 不在前端伪造构建结果
- 上传 request 参数与后端 contract 完整一致
- 表单、hook、API client 不得遗漏 `jurisdiction`、`domain` 等已公开字段

## 4.3 Chunk / Vector Inspector
审查重点：
- document / chunk / metadata 展示边界清楚
- build/version 信息可辨识
- 不直接操作底层数据库内部实现

## 4.4 Evaluation Dashboard
审查重点：
- 评估任务启动、等待、完成、失败状态明确
- retrieval / citation / answer 结果展示清楚
- 失败 case 可查看
- 不在前端重算后端评估逻辑

## 4.5 Runtime / Config View
审查重点：
- 配置展示字段准确
- 敏感信息不直接明文展示
- 当前阶段只读逻辑清楚

---

## 5. 组件与状态管理审查点

### 5.1 组件边界
禁止把整页逻辑全部写进一个巨大组件。  
页面应拆分出合理的：
- 输入区
- 结果区
- 状态区
- 错误区
- 辅助信息区

### 5.2 状态边界
禁止：
- 把全站状态无必要地全局化
- 把 streaming 状态和普通请求状态混用
- 为简单页面过早引入复杂状态框架

### 5.3 可测试性
核心交互逻辑应尽量可测试，尤其是：
- SSE 事件处理
- cancel 状态流转
- 错误状态展示
- API 响应映射
- request 参数从页面到 API client 的映射完整性

---

## 6. 当前阶段应拒绝的改动

### 6.1 应拒绝的越界改动
- 把控制台前端扩展成正式用户产品前端
- 在前端中实现后端核心业务规则
- 页面直接依赖后端内部实现而不是 API
- 直接访问 Qdrant 等底层组件作为主交互路径

### 6.2 应拒绝的结构性坏味道
- 巨型页面组件
- 大量 `any`
- 大量重复 API 请求逻辑
- 大量复制粘贴的 SSE 解析逻辑
- 通过硬编码假数据“假装页面完成”

### 6.3 应拒绝的交互性问题
- 错误不可见
- stream 状态不明确
- cancel 不生效但 UI 无提示
- 构建/评估进行中与完成态无区分

---

## 7. 当前阶段特别要求
- 后端控制面完成持久化升级后，控制台应优先跟随契约最小增量更新
- Chunk / Vector Inspector 中的“查看向量”应基于后端按 `vector_point_id` 回读 Qdrant 的能力

当前第一版控制台优先工程可用性，不以视觉包装为主要评审标准。

更看重：
- 是否能操作
- 是否能看见状态
- 是否能调试
- 是否能和后端稳定协作

不优先：
- 是否视觉炫丽
- 是否设计系统完整
- 是否商业化体验成熟

---

## 8. 文档与代码一致性要求
如果页面范围、后端依赖关系或前端边界发生变化，必须同步更新：
- `AGENTS.md`
- `PROJECT_PLAN.md`
- `ARCHITECTURE.md`
- `CODE_REVIEW.md`

禁止前端代码已经扩展，但治理文档仍停留在旧边界。
